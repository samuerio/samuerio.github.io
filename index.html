消灭CallBack导致的函数金字塔

对于开发人员来说，既要保持页面与用户的交互不受影响，又要协调页面与异步任务的关系，这种非线性的编程存在适应的困难。传统的编程方式是采用CallBack来协调「用户交互」和「异步任务」，但这种方式会导致嵌套的函数，不仅难以理解，而且开发人员需要仔细分析哪些代码用于应用的业务逻辑，而哪些代码处理异步函数调用，代码结构支离破碎，难以维护。

Promise模式拯救「延迟」引起CallBack

案例1

服务请求是有延迟、需要异步的，loadEvent是异步任务，但其中既有同步处理逻辑又有异步处理逻辑，不仅不好理解，而且难以查看服务请求来源（异步任务）。

优化

这里的话是将服务请求(异步任务)都归于service中，并使用Promise模式，返回的deferred用于同步地写入请求到数据后的处理。不仅消除了烦人的callback取而代之使用then，使得同步式编码更顺畅，而且更易于理解。(注意:callback会打断同步式编码，应该尽量避免)

事件注册模式拯救「用户交互」引起的CallBack

这种模式类似Dom和相关事件之间的关系，当Dom被点击，则发射出一个点击事件的’信号’，触发点击事件的监听等。

案例2

优化

事件机制：提前注册「文本信息接受事件」的监听程序，当接受到文本信息时，发出一个文本信息接受事件，调用监听程序。这种写法使得晦涩的回调函数消失代替以事件注册，增加了程序的可读性和灵活性。使用者使用封装的JS对象时，只要关注JS对象提供哪些事件点就好了，不需要关注其中的处理逻辑。

案例3

这种写法非常不推荐！节点渲染是一个「同步」过程，而事件程序的执行是「异步」的，它必须等待事件的发出。将同步和异步逻辑写在一起，不仅打断正常编码思路，而且也会导致后期维护起来异常困难，这里会导致：找节点点击事件监听程序（异步任务）时就不好找/(ㄒoㄒ)/~~

优化

这种写法，可以一目了然得知道listener是「用户操作」异步任务入口。分离了同步和异步的代码使得程序层次更加分明，而且可读性和可维护性大大提高！

Content-type决定了文件类型以及数据的展示方式

The purpose of the Content-Type field is to describe the data contained in the body fully enough that the receiving user agent can pick an appropriate agent or mechanism to present the data to the user, or otherwise deal with the data in an appropriate manner.
content-type

Get请求的数据是附着在URL后（即位于HTTP协议头部），而Post提交的数据是放于HTTP包的包体。故content-type只会影响到Post提交的数据。

application/x-www-form-urlencoded

application/x-www-form-urlencoded为「表单提交」，默认的Ajax Post提交也是按这种方式。

application/json;charset=utf-8

这种时候，只能从HTTP内容体里面获取提交的数据

同步异步与阻塞非阻塞

同步与异步

同步和异步关注的是消息通信机制

同步：调用发出后，没得到结果前，该调用就不返回。
异步：调用发出后，调用就直接返回了(没有返回结果)。调用者不会立即得到结果，而是在调用发出后，被调用者通过状态、通知来通知调用者或通过回调函数来处理这个调用。

阻塞与非阻塞同步异步和阻塞非阻塞

阻塞和非阻塞关注的是程序在等待调用结果(消息，返回值)时的状态。

阻塞：调用结果返回前，线程会被挂起
非阻塞：不能得到结果之前，该调用不会阻塞当前线程
